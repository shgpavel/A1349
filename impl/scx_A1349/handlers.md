# Спецификация обработчиков sched_ext для s4+ (A1349)

Данный документ описывает отображение логики чистой аукционной модели (без виртуального времени) на функции обратного вызова (operations) подсистемы `sched_ext`.

## 1. `scx_ops.enqueue`

Вызывается ядром Linux, когда задача готова к выполнению.

**Шаги в s4+:**
1.  Вычисление длины задачи $l_i$: берётся EWMA длительности задачи (`len_est_ns`), делится на `AUCTION_SLICE_P`.
2.  Вычисление $\phi_P(i)$ и $\phi_E(i)$ на основе $l_i$ и веса задачи.
3.  Определение предпочитаемого кластера: $\kappa^* = \arg\max_\kappa \phi_\kappa$.
4.  Вставка задачи в соответствующее дерево (BPF RB-Tree или приоритетную очередь, эмулирующую DSQ), где ключом является значение $\phi_{\kappa^*}$.
5.  Вызов `scx_bpf_kick_cpu` для простаивающего ядра кластера $\kappa^*$, чтобы сохранить свойство `work-conserving`.

*Внимание: `enqueue` не занимается вычислением VCG-платежей, так как платёж зависит от состояния очереди в момент извлечения задачи.*

## 2. `scx_ops.dispatch`

Вызывается, когда ядро типа $\kappa$ освобождается и запрашивает задачу для выполнения.
Здесь выполняется основной аукционный расчёт.

**Шаги в s4+:**
1.  **Peek (Извлечение top-1 и top-2):**
    Планировщик заглядывает в приоритетную структуру (BPF RB-Tree) для кластера $\kappa$.
    Извлекается лучшая задача ($i^*$) и читается ценность второй по приоритету задачи ($j$). Если очередь пуста, переход к кросс-кластерному заимствованию.
2.  **Вычисление VCG-платежа:**
    $$p_{i^*} = \phi_\kappa(j) + (\delta^{m_\kappa(l_j)} - \delta^{m_\kappa(l_{i^*})}) \bar{W}_\kappa$$
    Значения $\delta^m$ извлекаются из предвычисленной таблицы `delta_table`. $\bar{W}_\kappa$ берётся из `auction_ctx`.
3.  **Проверка бюджета (Budget check):**
    *   Если $p_{i^*} \le B_{i^*}^t$ (`budget`):
        Бюджет задачи уменьшается на $p_{i^*}$. Задача $i^*$ удаляется из RB-Tree и передаётся в `SCX_DSQ_LOCAL` ядра через вызов `scx_bpf_dsq_insert`.
    *   Если $p_{i^*} > B_{i^*}^t$:
        Задача не может оплатить аукцион. Она перемещается в `AUCTION_DSQ_STARVED`. Ядро повторяет процесс для задачи $j$ (которая становится новым $i^*$).

## 3. `scx_ops.stopping`

Вызывается, когда задача снимается с выполнения (завершение кванта, вытеснение, блокировка).

**Шаги в s4+:**
1.  Сбор статистики времени выполнения: вычисление фактически использованного времени `t_consumed`.
2.  Обновление EWMA длительности задачи `len_est_ns`, используемого как прокси для $l_i$.
3.  **Обновление $\bar{W}_\kappa$ (обучение механизма):**
    Добавление реализованного вклада задачи $\phi_\kappa(i^*) \cdot (t_{consumed} / q)$ в общую оценку будущего блага $\bar{W}_\kappa$ по формуле EWMA с $\alpha = 1/16$.

*Внимание: в отличие от `s4` и EEVDF, здесь не происходит никакого пересчёта системного виртуального времени $V(t)$ или дедлайнов $ve/vd$.*

## 4. `scx_ops.enable` и `scx_ops.disable`

**enable:**
*   Инициализация `task_ctx`.
*   Назначение стартового бюджета $B_i = w_i \cdot \text{BUDGET\_MUL}$.

**disable:**
*   Освобождение памяти, очистка статистики.
*   Никаких коррекций "суммарного отставания", так как понятие `lag` в чистой модели отсутствует.

## 5. `scx_ops.set_weight`

Вызывается при изменении `nice`-значения задачи или её cgroup-веса.
Обновляет базовую полезность $v_i$, максимальный бюджет и пропорционально масштабирует остаточный бюджет задачи. Пересчёт $V(t)$ не требуется.

## 6. Механизм кросс-кластерного заимствования (Опционально, для work-conservation)

Если кластер P пуст, а ядро P освободилось, оно может попытаться извлечь задачу из `AUCTION_DSQ_E`. 
Поскольку $j$ находится в очереди E, её ценность была вычислена для E-ядра. При заимствовании на P-ядро ценность должна быть пересчитана, или планировщик может использовать упрощённый fallback: извлечь $i^*$ с оплатой резервной цены без вычисления маргинальной разницы $\phi_E(j)$ для $P$-ядра.
