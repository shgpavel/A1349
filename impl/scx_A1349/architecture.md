# Архитектура аукционного планировщика s4+ (A1349)

Данный документ описывает внутреннее устройство eBPF-планировщика `scx_s4plus` для подсистемы `sched_ext`. В отличие от предыдущей версии `s4`, реализация `s4+` полностью отказывается от виртуального времени `ve`/`vd` и EEVDF-очередей, переходя к чистой аукционной модели на базе маргинальных VCG-платежей, как это описано в математической модели A1349.

## 1. Структура очередей (Direct Auction Queues)

В планировщике используются две кластерные очереди и одна вспомогательная:

*   `AUCTION_DSQ_P` — очередь для ядер высокой производительности (P-cores).
*   `AUCTION_DSQ_E` — очередь для энергоэффективных ядер (E-cores).
*   `AUCTION_DSQ_STARVED` — очередь для задач, у которых остаточный бюджет ($B_i^t$) не позволяет оплатить текущий VCG-платёж. Эти задачи ждут пополнения бюджета за счёт простоя.

Обе основные очереди (`DSQ_P` и `DSQ_E`) используют BPF RB-Tree или другую структуру с приоритетом для хранения задач. Ключом для сортировки выступает не виртуальный дедлайн, а **эффективная ценность задачи** $\phi_\kappa(\theta_i)$.

### Вычисление эффективной ценности $\phi_\kappa(\theta_i)$

При поступлении задачи (в обработчике `enqueue`) вычисляются две оценки ценности:

*   На P-ядре: $\phi_P(i) = v_i - c_P \cdot l_i$
*   На E-ядре: $\phi_E(i) = v_i - c_E \cdot \lceil l_i \cdot \sigma \rceil$

Где:
*   $v_i$ — базовая полезность задачи, аппроксимируемая весом `p->scx.weight`.
*   $l_i$ — ожидаемая длина задачи в квантах ($q$). В реализации вычисляется как EWMA-оценка времени непрерывной работы задачи, разделённая на номинальный квант `AUCTION_SLICE_P`.
*   $\sigma = \eta_P / \eta_E$ — соотношение производительности P и E ядер.

Задача помещается в ту очередь $\kappa$, где её $\phi_\kappa$ максимален ($\kappa^* = \arg\max_\kappa \phi_\kappa$). Ключом сортировки выступает $\phi_{\kappa^*}$, причём извлечение происходит в порядке убывания $\phi$.

## 2. Переменные состояния и BPF Maps

```c
// Состояние конкретной задачи (task_storage)
struct task_ctx {
    u64 v_i;              // Полезность задачи (вес * множитель)
    u64 len_est_ns;       // EWMA длительности исполнения (в нс)
    u64 budget;           // Текущий остаточный бюджет B_i^t
    u64 budget_max;       // Максимальный бюджет (B_i)
    int on_p_type;        // Кластер, в котором задача находится в данный момент
};

// Глобальное состояние планировщика (ARRAY map)
struct auction_ctx {
    u64 w_bar_p;          // \bar{W}_P: Ожидаемое будущее благосостояние для P-ядра
    u64 w_bar_e;          // \bar{W}_E: Ожидаемое будущее благосостояние для E-ядра
    
    u32 max_capacity;     // \eta_P
    u32 min_capacity;     // \eta_E
    u32 cost_p;           // c_P
    u32 cost_e;           // c_E
    
    // Lookup-таблица степеней дисконтирования \delta (с фиксированной точкой)
    // delta_table[m] хранит \delta^m * SCALE
    u64 delta_table[MAX_CONTRACT_LENGTH];
};
```

## 3. Расчёт VCG-платежей

Ключевым отличием `s4+` является точный расчёт VCG-платежа в момент выбора задачи на исполнение (dispatch). Когда ядро типа $\kappa$ освобождается, оно берёт из своей очереди две самые приоритетные задачи: победителя $i^*$ и следующего кандидата $j$ (runner-up).

VCG-платёж $p_{i^*}$ рассчитывается по формуле:

$$ p_{i^*} = \phi_\kappa(\theta_j) + (\delta^{m_\kappa(l_j)} - \delta^{m_\kappa(l_{i^*})}) \bar{W}_\kappa $$

**Компоненты расчёта:**
1.  **$\phi_\kappa(\theta_j)$**: Экстерналия, налагаемая на конкурента. Получается путём чтения (peek) второй задачи в очереди.
2.  **$m_\kappa(l)$**: Длительность контракта в квантах. $m_P = \lceil l_{i^*} / q \rceil$, $m_E = \lceil l_{i^*} \cdot \sigma / q \rceil$. Ограничивается константой `MAX_CONTRACT_LENGTH` для использования в lookup-таблице `delta_table`.
3.  **$\bar{W}_\kappa$**: Обновляется при каждом завершении кванта (в `stopping`) как скользящее среднее (EWMA) от реализованных в системе эффективных ценностей $\phi$ на соответствующем кластере.

## 4. Бюджетная дисциплина (Budget Check)

Поскольку `sched_ext` не позволяет "заглядывать" (peek) в стандартный `scx_bpf_dsq` без извлечения, `s4+` использует BPF-коллекцию (например, собственное BPF RB-Tree) для диспетчеризации. Это позволяет получить доступ к $j$.

Если VCG-платёж $p_{i^*}$ превышает остаточный бюджет задачи `task_ctx->budget`, задача не назначается на ядро. Вместо этого:
1. Она перемещается в очередь `AUCTION_DSQ_STARVED`.
2. Ядро переходит к следующему кандидату $j$.
3. Бюджет задачи в `STARVED` восполняется, пока задача спит или ждёт в очереди (аналогично механизму `token bucket`).

Если бюджет позволяет, из `budget` списывается сумма $p_{i^*}$, и задача переносится в `SCX_DSQ_LOCAL` текущего ядра для исполнения.

## 5. Обновление $\bar{W}_\kappa$

При завершении кванта (обработчик `stopping`) планировщик обновляет оценку будущей ценности $\bar{W}_\kappa$.

```c
W_realized = phi_k(i) * (t_consumed / q); // Вклад задачи за квант
W_bar_k = (W_bar_k * 15 + W_realized) / 16;  // EWMA с alpha = 1/16
```
Этот параметр передаётся в следующий расчёт VCG-платежа, обеспечивая адаптацию механизма к изменяющемуся потоку задач.
